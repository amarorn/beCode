# Topologia por Ambiente — IDE Corporativa de Dados

**Task:** 1.3 — Definir arquitetura de entrega e ambientes
**Versão:** 1.0.0
**Data:** 2026-04-18
**Status:** Rascunho (premissas explícitas; revisar após conclusão da task 1.2 — C4)

---

## 1. Premissas

- Cloud alvo: **GCP** como primário (justificativa: integração nativa com BigQuery, que é integração obrigatória da task 9.1). Databricks como workload externo federado.
- Kubernetes gerenciado (**GKE**) para control plane e execution plane.
- Control plane e execution plane em **clusters separados** (regra arquitetural da task 1.2).
- Multi-tenant lógico no control plane; isolamento físico por namespace + NetworkPolicy no execution plane.
- Região primária: `southamerica-east1` (São Paulo). DR em `us-central1`.

## 2. Ambientes

| Ambiente | Propósito | Isolamento | Dados |
|---|---|---|---|
| **dev** | Desenvolvimento interno da plataforma | Projeto GCP dedicado | Sintéticos |
| **staging** | Homologação com stakeholders e testes de integração | Projeto GCP dedicado, rede isolada | Amostras anonimizadas |
| **prod** | Produção para tenants corporativos | Projeto GCP dedicado, VPC-SC | Dados reais, classificados |
| **dr** | Disaster recovery ativo-passivo de prod | Região secundária | Replicação assíncrona |

**Regra:** promoção só via pipeline (`dev → staging → prod`); nenhum acesso humano direto de escrita a prod.

## 3. Blocos por Ambiente

### 3.1 Control Plane (por ambiente)

| Componente | Tecnologia | Dimensionamento inicial (prod) |
|---|---|---|
| API Gateway / Ingress | GKE Ingress + Cloud Armor | 2 réplicas, HPA 2–10 |
| Serviços de domínio (Identity, AI Gateway, Policy, Workspaces, Extensions, Audit) | GKE Deployments (Go/Node) | 2 réplicas cada, HPA por CPU/req |
| Banco relacional primário | Cloud SQL PostgreSQL 16, HA regional | db-custom-4-16384 |
| Cache / filas leves | Memorystore Redis (HA) | 5 GB |
| Mensageria / eventos | Pub/Sub | Tópicos por domínio |
| Object storage | GCS (buckets por domínio, CMEK) | — |
| Logs de auditoria imutáveis | GCS Bucket Lock + BigQuery sink | Retenção 7 anos |

### 3.2 Execution Plane (workspaces)

| Componente | Tecnologia | Notas |
|---|---|---|
| Cluster de workspaces | GKE dedicado (node pools por classe de workload) | `standard`, `gpu-opt`, `mem-opt` |
| Workspace pod | Container por usuário/sessão | Quotas CPU/mem/disco; NetworkPolicy restritiva |
| Registry de imagens | Artifact Registry (CMEK) | Imagens base assinadas |
| Terminal proxy | Serviço no control plane | Proxy WebSocket com policy engine inline |

### 3.3 Ingress e rede

- Ingress externo: Cloud Load Balancer + Cloud Armor (WAF, rate limiting por IP/tenant).
- mTLS entre serviços via **service mesh leve** (Anthos Service Mesh ou Linkerd — decisão ADR-0002, pendente).
- Segmentação: `control-plane-subnet`, `execution-plane-subnet`, `data-egress-subnet`.
- Egress de workspaces para BigQuery/Databricks somente via **Private Service Connect** / IPs autorizados (lista branca).

## 4. Matriz de Dimensionamento Inicial

| Recurso | dev | staging | prod |
|---|---|---|---|
| Nodes control plane | 3 × e2-standard-4 | 3 × e2-standard-8 | 6 × e2-standard-8 (autoscale → 20) |
| Nodes execution plane | 2 × e2-standard-4 | 3 × e2-standard-8 | 3–30 × e2-standard-16 (autoscale) |
| Cloud SQL | db-custom-1-3840 | db-custom-2-8192 HA | db-custom-4-16384 HA |
| Redis | 1 GB | 3 GB | 5 GB HA |
| Limite de workspaces simultâneos | 20 | 100 | 1.000 (inicial) |

**Risco conhecido:** subdimensionamento de workspaces. Mitigação: Cluster Autoscaler + quotas por tenant + alerta em 70% de saturação do node pool.
