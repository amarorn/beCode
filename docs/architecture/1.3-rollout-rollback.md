# Estratégia de Rollout e Rollback — IDE Corporativa de Dados

**Task:** 1.3 — Definir arquitetura de entrega e ambientes
**Versão:** 1.0.0
**Data:** 2026-04-18
**Status:** Rascunho

---

## 1. Objetivo

Entregar software em produção de forma **reversível, observável e segura**, minimizando blast radius de falhas.

## 2. Estratégia por tipo de componente

| Componente | Estratégia de rollout | Rollback |
|---|---|---|
| Serviços stateless (control plane) | **Canary progressivo** (5% → 25% → 50% → 100%) via Argo Rollouts | Reapontar tag para versão anterior (< 5 min) |
| Terminal proxy / Policy engine | **Blue/Green** — corte só após validação de políticas | Switch de tráfego de volta ao blue |
| Workspace image base | **Shadow** — imagens novas coexistem; workspaces novos usam nova tag | Tag anterior permanece publicada; downgrade por policy |
| Migrações de banco | **Expand → Migrate → Contract**; migrações forward-only em prod | Compensação por nova migration; nunca `DROP` na mesma release |
| Extensões corporativas | Publicação no catálogo + flag por tenant | Despublicar versão; tenants pinados à anterior |
| Modelos/providers LLM no AI Gateway | **Feature flag por tenant** (roll-forward) | Desabilitar provider via config; requests fazem fallback |

## 3. Pipeline de promoção

```
build (CI) → scan (SAST, dep, container) → test (unit + integration) →
  publish artifact (immutable SHA tag) →
    auto-deploy DEV →
      smoke tests DEV (gate) →
        auto-deploy STAGING →
          regression + smoke STAGING (gate) →
            manual approval (dupla) →
              canary PROD (5%) →
                auto-analysis (SLO) →
                  progressão ou abort automático
```

**Regras:**
- Nenhum artefato é reconstruído entre ambientes. Promoção = mover tag/ref.
- Cada step emite evento para o Audit Log imutável (task 11.2).

## 4. Critérios de avanço do canary

Avaliados a cada fase por **5 min** antes de progredir:

| Métrica | Limite para abort |
|---|---|
| Error rate HTTP 5xx | > 1% acima do baseline |
| p95 de latência | > 20% acima do baseline |
| Falhas de policy engine | qualquer regressão |
| Saturação de CPU/mem dos pods | > 85% sustentado |

Abort → rollback automático para versão anterior; incidente aberto.

## 5. Rollback

- **RTO alvo:** < 5 min para serviços stateless; < 30 min para componentes com estado.
- **RPO alvo:** ≤ 15 min (replicação Cloud SQL + backups PITR).
- **Mecânica:**
  - Stateless: `argo rollouts undo` ou reapontamento de imagem.
  - DB: nunca drop; rollback via nova migration de compensação; restore PITR apenas em incidente sev-1.
  - Config: mudanças GitOps revertidas por `git revert` + sync automático.

## 6. Releases e janelas

- **Janela padrão:** terça e quarta, 10h–16h BRT.
- **Code freeze:** 24 h antes de eventos corporativos relevantes.
- **Hotfix:** fora de janela, com aprovação do on-call lead + registro de incidente.

## 7. Smoke e verificação

Cada deploy executa suite de smoke contra o ambiente alvo:

- Login SSO end-to-end.
- Criar e derrubar workspace (happy path).
- Executar query no BigQuery de teste com estimativa de custo.
- Chamada ao AI Gateway com prompt controlado.
- Escrita e leitura em Audit Log.

Falha em qualquer passo → rollback automático.

## 8. Failover e DR

- **Failover básico:** Cloud SQL HA (regional) promove réplica automaticamente.
- **DR regional:** runbook documentado; exercício trimestral; replicação assíncrona para `us-central1`.
- **Rollback simulado:** exercício mensal em staging; relatório publicado.
