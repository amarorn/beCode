**Task:** 1.3 — Definir arquitetura de entrega e ambientes
**Versão:** 1.0.0
**Data:** 2026-04-18
**Status:** Rascunho

## 1. Princípios

1. **Segmentação** entre control plane, execution plane e egress.
2. **Deny-by-default** em egress de workspaces; lista branca explícita.
3. **Zero confiança entre serviços**: mTLS obrigatório, identidade por workload (Workload Identity).
4. **Nenhum IP público nos workloads**; exposição externa apenas via Ingress gerenciado.
5. **Observabilidade de rede** ativada em todos os segmentos (VPC Flow Logs, Firewall Insights).

## 2. Topologia de VPC (por ambiente)

Uma VPC por ambiente (`becode-dev`, `becode-staging`, `becode-prod`), sem peering entre ambientes.

| Subrede | CIDR exemplo (prod) | Uso | Egress externo |
|---|---|---|---|
| `subnet-ctl-plane` | `10.10.0.0/20` | Nodes GKE control plane | Não (NAT controlado) |
| `subnet-exec-plane` | `10.10.16.0/20` | Nodes GKE execution plane (workspaces) | Via PSC / NAT restrito |
| `subnet-data-egress` | `10.10.32.0/24` | Saída dedicada para BigQuery/Databricks | Via Private Service Connect |
| `subnet-svc` | `10.10.33.0/24` | Cloud SQL, Redis (private service access) | Não |
| `subnet-mgmt` | `10.10.34.0/24` | Bastion/IAP, tooling | Restrito a IP corporativo |

- **Secondary ranges** para Pods/Services em `subnet-ctl-plane` e `subnet-exec-plane` (alias IPs).
- **Private Google Access** habilitado em todas.

## 3. Ingress

| Camada | Tecnologia | Função |
|---|---|---|
| Edge | Cloud Load Balancer externo + **Cloud Armor** | WAF, rate limiting por IP, geo-fencing |
| Identity-aware | IAP (para consoles internos) | Acesso a ferramentas internas sem VPN |
| Ingress do app | GKE Ingress (HTTPS, TLS 1.3, HSTS) | Terminação TLS, roteamento |
| Mesh interno | ASM/Linkerd (mTLS) | Identidade por serviço; autorização L7 |

TLS: certificados gerenciados pelo Google, rotação automática. Pins internos via cert-manager + CA privada.

## 4. Egress

### 4.1 Control plane

- Cloud NAT dedicado com IPs fixos (para allowlists de parceiros).
- Saída permitida apenas para: APIs Google, provedores de LLM homologados (via AI Gateway), domínios de observabilidade.

### 4.2 Execution plane (workspaces)

- **Default deny.** Pod sem NetworkPolicy não tem egress.
- Destinos permitidos por padrão:
  - BigQuery, Cloud Storage (via **PSC**, sem internet).
  - AI Gateway (ClusterIP interno).
  - Artifact Registry (imagens base).
  - Package registries corporativos (PyPI mirror, npm mirror).
- **Destinos externos** (GitHub, Databricks, APIs de clientes) requerem **allowlist por tenant**, aprovada e auditada.
- Terminal proxy injeta policy engine; comandos que tentam rotas não permitidas falham antes do syscall.

## 5. Integrações privadas

| Integração | Mecanismo |
|---|---|
| BigQuery | Private Service Connect endpoint por ambiente |
| Cloud SQL | Private Service Access (sem IP público) |
| Databricks | **VPN HA** ou PSC, conforme conta Databricks do cliente |
| dbt Cloud / Dataform | HTTPS via NAT com IPs fixos em allowlist |
| SSO IdP corporativo | HTTPS saída via NAT fixo + allowlist no IdP |

## 6. Firewall e NetworkPolicy

- **Regras VPC:** negar tudo por padrão; permitir por tag de serviço.
- **NetworkPolicy Kubernetes:** obrigatória para todo Namespace de tenant. Template baseline:
  - allow `ingress` somente do Ingress Gateway do mesh.
  - allow `egress` para DNS interno, AI Gateway, PSC endpoints.
  - deny resto.
- **Host firewall** nos nodes: somente portas do kubelet e do mesh expostas a control plane interno.

## 7. DNS

- Zona privada `becode.internal` resolvida somente dentro das VPCs.
- Split-horizon: nomes públicos só para edge endpoints (`app.becode.example.com`).
- DNS queries de workspaces vão para resolver gerenciado com logs; queries para domínios em blocklist retornam NXDOMAIN.

## 8. Observabilidade de rede

- **VPC Flow Logs** com 100% sampling em prod (retenção 30 dias, export BigQuery por 1 ano).
- **Firewall Insights** diário.
- Alertas: tentativas de egress bloqueadas > threshold por tenant; novas FQDNs em requests DNS.

## 9. Pendências

- Dimensionar NAT (conexões simultâneas) conforme projeção de workspaces concorrentes.
- Definir quem aprova allowlists externas por tenant (provavelmente SRE + Security).
