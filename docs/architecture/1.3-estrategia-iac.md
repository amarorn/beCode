## 1. Princípios

1. **Tudo versionado.** Nenhuma mudança em infra ou config aplicada fora do repositório.
2. **Segredos nunca no código.** `git-secrets` + scanner em PR; referência por nome, valor em Secret Manager.
3. **Deploy imutável.** Imagens versionadas por SHA; rollback por reapontamento de tag, nunca por mutação.
4. **Ambientes isolados.** Um projeto GCP por ambiente; credenciais e state separados.
5. **Promoção por pipeline, não por humano.** Aprovações registradas; nenhum `terraform apply` local em staging/prod.

## 2. Stack

| Camada | Ferramenta | Justificativa |
|---|---|---|
| Provisionamento de nuvem | **Terraform 1.8+** | Maturidade, módulos GCP oficiais, ecossistema |
| Estado remoto | GCS bucket por ambiente + state lock | Isolamento; auditoria |
| Kubernetes apps | **Helm 3** + **Argo CD** (GitOps) | Declarativo; reconciliação contínua |
| Secrets | **Google Secret Manager** + External Secrets Operator | Fora do código; sincronização controlada |
| Policy-as-code | **OPA / Conftest** em CI | Barrar drift e violações antes do apply |
| Scan de IaC | **tfsec** + **Checkov** | Segurança baseline |

## 4. Fluxo de mudança

```
PR → lint/fmt → tfsec + Checkov → Conftest (policies) →
  terraform plan (dev/staging/prod) postado no PR →
    review obrigatório (1 reviewer para dev; 2 para prod) →
      merge → pipeline aplica em dev →
        promote manual para staging →
          promote com aprovação dupla para prod
```

- **Apply em prod** requer: janela de mudança aprovada, segundo aprovador, plan re-executado com hash idêntico ao aprovado.
- **Drift detection:** job diário `terraform plan` em cada ambiente; divergência abre incidente.

## 5. Gestão de segredos

- Criados e rotacionados via Secret Manager; **nunca** saem como output de Terraform.
- Workloads consomem via External Secrets Operator (sync para `Secret` Kubernetes em namespaces autorizados).
- Rotação: chaves CMEK anualmente; credenciais de integração a cada 90 dias.
- Pre-commit hook + CI scanner (`gitleaks`) bloqueiam push com padrão de segredo.

## 6. Convenções

- Versionamento de módulos: SemVer; tag `modules/gke-control-plane/v1.3.0`.
- Naming: `<projeto>-<ambiente>-<recurso>-<n>`, ex: `becode-prod-gke-ctl-01`.
- Tags obrigatórias em todo recurso: `env`, `owner`, `cost-center`, `data-class`.
- State lock via GCS + Cloud Storage object versioning (recovery de state corrompido).
