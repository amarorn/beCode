# Mapa de Jornadas — IDE Corporativa para Engenharia de Dados

**Task:** 1.1 — Definir visão do produto, personas e boundaries
**Versão:** 1.0.0
**Data:** 2026-04-18

> Mapeia as jornadas críticas das personas definidas em `1.1-visao-produto.md`. Cada jornada descreve **fases → ações → pontos de dor atuais → expectativa com a IDE → oportunidades**.

---

## J1 — Onboarding de novo engenheiro de dados (Dani)

| Fase | Ação hoje | Dor atual | Expectativa IDE | Oportunidade |
|------|-----------|-----------|-----------------|--------------|
| Acesso | Solicita credenciais via tickets | 2–5 dias de espera | SSO/OIDC + provisionamento automático | Onboarding < 1h |
| Setup | Instala Python, dbt, gcloud, clona repo | Divergência entre máquinas | Workspace template pronto | "One-click workspace" |
| Primeira execução | Roda primeiro modelo dbt | Falta dados de amostra | Dataset seed + preview | Tutorial guiado |
| Primeiro commit | Abre PR | Não sabe padrões do squad | Templates + lint embarcado | TTFC < 30 min |

**Métrica chave:** TTFC mediano.

---

## J2 — Desenvolver um pipeline dbt + BigQuery (Dani / Bea)

| Fase | Ação | Dor | Expectativa | Oportunidade |
|------|------|-----|-------------|--------------|
| Explorar dados | Abre BigQuery console separado | Troca de contexto | Data explorer embutido | Preview lado-a-lado do SQL |
| Escrever modelo | Edita `.sql` no VS Code | Sem autocomplete de schema | LSP dbt + schema-aware | Autocomplete + lineage inline |
| Validar | Executa `dbt run --select` local | Custo BigQuery invisível | *Dry-run* + estimativa de custo | Guardrail antes de executar |
| Usar IA | Copia trecho para ChatGPT pessoal | Risco de vazamento | IA contextual governada | Prompt policy aplicada |
| Commitar | `git commit` via terminal local | Hooks inconsistentes | Terminal governado + hooks oficiais | Policy engine |

**Métrica chave:** % sessões que usam IA, custo por execução.

## J3 — Revisão de PR e aplicação de padrões (Rafa)

| Fase | Ação | Dor | Expectativa | Oportunidade |
|------|------|-----|-------------|--------------|
| Revisão | Abre PR no GitHub | Falta contexto de execução | Link do workspace + logs anexos | "Reproduce in IDE" |
| Padrões | Comenta manualmente | Repetitivo | Lint corporativo + templates | Redução de lead time |
| Métricas | Consulta múltiplas ferramentas | Fragmentação | Dashboard de adoção | Telemetria unificada |

**Métrica chave:** lead time de PR; % squads usando templates.

## J4 — Resposta a incidente de segurança (Sam)

| Fase | Ação | Dor | Expectativa | Oportunidade |
|------|------|-----|-------------|--------------|
| Detecção | Recebe alerta | Logs dispersos | Audit log imutável central | Correlação por tenant |
| Investigação | Tenta reconstruir sessão | Impossível | Trace OTel por sessão/prompt | Forensics estruturado |
| Contenção | Revoga acessos manualmente | Lento | RBAC + kill-switch workspace | Resposta em minutos |
| Post-mortem | Escreve documento | Sem dados estruturados | Export de eventos | Evidências auditáveis |

**Métrica chave:** MTTR de incidentes; 0 vazamentos via LLM.

## J5 — Publicar extensão corporativa (Rafa)

| Fase | Ação | Dor | Expectativa | Oportunidade |
|------|------|-----|-------------|--------------|
| Desenvolver | Cria extensão local | Sem SDK oficial | SDK + template de extensão | Paridade com VS Code API |
| Submeter | Processo manual por e-mail | Sem rastreio | Portal de submissão | Workflow de aprovação |
| Aprovar | Revisão ad-hoc | Critérios inconsistentes | Checklist de segurança automatizado | Governança clara |
| Publicar | Distribuição manual | Instalação divergente | Catálogo interno | Adoção rastreável |

**Métrica chave:** nº de extensões aprovadas/mês; % workspaces com extensões oficiais.

## J6 — Uso diário de IA no código (Dani, Bea)

| Fase | Ação | Dor | Expectativa | Oportunidade |

| Invocar IA | Abre ChatGPT externo | Contexto manual, risco | Side-panel contextual | Repo-aware prompts |
