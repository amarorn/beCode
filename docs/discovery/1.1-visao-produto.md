**Task:** 1.1 — Definir visão do produto, personas e boundaries
**Versão:** 1.0.0
**Data:** 2026-04-18
**Status:** Rascunho para aprovação dos stakeholders

---

## 1. Visão

> Oferecer uma **IDE corporativa web-first** para engenharia de dados que una, em um único ambiente governado, o desenvolvimento (SQL, Python, dbt/Dataform), a execução controlada em workspaces isolados, a integração nativa com BigQuery/Databricks e uma camada de IA multi-LLM com políticas de custo, segurança e auditoria — reduzindo o tempo entre *ideia → pipeline em produção* e eliminando o uso fragmentado de IDEs locais sem governança.

### 1.1 Proposta de valor

- **Para o engenheiro de dados:** ambiente pronto em minutos, com conectores, templates e IA contextual aprovada pela empresa.
- **Para a liderança técnica:** padronização de stack, rastreabilidade de execuções e custo de LLM/consultas atribuído por squad.
- **Para segurança/compliance:** deny-by-default no terminal, mascaramento de segredos, auditoria imutável e catálogo curado de extensões.

### 1.2 Diferenciais estratégicos

1. **Governança nativa de IA** — prompt policy, rate limiting e atribuição de custo por tenant/squad.
2. **Policy engine no terminal** — execução corporativa segura, diferente de IDEs locais.
3. **Integrações de dados first-class** — BigQuery, Databricks, dbt/Dataform com *cost guardrails*.
4. **Catálogo corporativo de extensões** com workflow de aprovação.
5. **Control plane separado do execution plane** — sem lock-in de LLM nem de registry.

## 2. Personas

### 2.1 Persona primária — Engenheiro(a) de Dados Pleno/Sênior ("Dani")

- **Contexto:** trabalha em squad de dados, mantém pipelines dbt e jobs PySpark/BigQuery.
- **Ferramentas atuais:** VS Code + múltiplas extensões, PyCharm em casos isolados, terminal local, credenciais espalhadas.
- **Dores:**
  - Setup de ambiente demora horas/dias; divergência entre máquinas.
  - Uso de LLMs pessoais (ChatGPT/Copilot) sem aprovação, com risco de vazamento.
  - Falta de visibilidade do custo de queries em BigQuery antes de executar.
- **Jobs to be done:**
  - *When* preciso prototipar um pipeline, *I want* ambiente pronto com credenciais e dados de amostra, *so I can* focar na modelagem.
  - *When* uso IA, *I want* que o contexto do repo seja enviado com segurança, *so I can* confiar nas sugestões.
- **Métricas de sucesso:** TTFC (time-to-first-commit) < 30min; NPS > 40.

### 2.2 Persona secundária — Tech Lead / Arquiteto(a) de Dados ("Rafa")

- **Contexto:** responsável por padrões técnicos, revisão de PRs e onboarding.
- **Dores:** ausência de templates oficiais, dificuldade de aplicar padrões entre squads, revisões lentas.
- **Jobs to be done:** publicar templates; exigir extensões aprovadas; medir adoção.
- **Métricas de sucesso:** % de squads usando templates oficiais; redução de *lead time* de PR.

### 2.3 Persona secundária — Analista de Dados/Analytics Engineer ("Bea")

- **Contexto:** escreve principalmente SQL e modelos dbt; baixa familiaridade com containers.
- **Dores:** barreira de entrada de IDEs técnicas; terminal assusta; falta de preview de dados.
- **Jobs to be done:** editar SQL com autocomplete de schema, rodar *dry-run* BigQuery, usar IA para documentar modelos.
- **Métricas de sucesso:** % de modelos documentados; ativação semanal.

### 2.4 Persona de governança — Segurança/Plataforma ("Sam")

- **Contexto:** define policies de execução, audita, responde a incidentes.
- **Dores:** ausência de trilha de auditoria unificada; risco de *prompt injection* e vazamento de PII em LLMs.
- **Jobs to be done:** configurar deny-lists; revisar logs imutáveis; aprovar extensões.
- **Métricas de sucesso:** 100% de comandos sensíveis auditados; 0 incidentes de vazamento por LLM.

### 2.5 Não-personas (quem **não** atendemos no MVP)

- Desenvolvedores full-stack/mobile.
- Cientistas de dados focados em treinamento de modelos em GPU.
- Usuários individuais fora de contexto corporativo (sem tenant/SSO).

## 4. Boundaries (In-Scope / Out-of-Scope do MVP)

### 4.1 In-Scope (MVP)

- Shell web dark-mode-first baseado em *editor engine* existente (ex.: Monaco/Code-OSS), **não próprio**.
- Autenticação SSO/OIDC e RBAC com escopo tenant/squad.
- Workspaces containerizados com templates.
- Terminal integrado com policy engine deny-by-default.
- AI Gateway multi-LLM com prompt policy, rate limit e atribuição de custo.
- Integrações: BigQuery, dbt, Git.
- Catálogo corporativo de extensões com aprovação manual.
- Observabilidade: OpenTelemetry, audit log imutável, telemetria de produto.

### 4.2 Out-of-Scope (explicitamente fora do MVP)

- **Editor engine próprio** (reafirmado pela regra do SpecDrive).
- IDE desktop instalável (apenas web no MVP).
- Marketplace público de extensões (apenas corporativo).
- Treinamento de modelos de IA / fine-tuning interno.
- Ambiente de produção gerenciado de jobs (*não é orquestrador*, integra com Airflow/Dataform/Databricks Jobs).
- Suporte offline.
- Compatibilidade com plugins PyCharm (**premissa a validar** — ver Riscos).
- Colaboração síncrona tipo *pair programming* em tempo real (Fase 2).
- Mobile.

### 4.3 Fase 2 (pós-MVP)

- Integração Databricks completa (notebooks, clusters).
- Debugger remoto.
- Colaboração síncrona.
- Extensões de terceiros via marketplace curado.
- Data catalog embutido.

## 6. Riscos e Premissas

| Risco | Impacto | Mitigação |
|-------|---------|-----------|
| Escopo excessivo no MVP | Atraso e burnout | Aplicar regra *"feature só entra com aderência a persona"*; revisão trimestral do backlog. |
| Premissa errada sobre compatibilidade PyCharm | Retrabalho e frustração de usuários migrando | Validar com entrevistas; posicionar explicitamente como **não** substitui PyCharm no MVP. |
| Adoção baixa por resistência a migrar do VS Code local | Falha de adoção | Paridade funcional mínima + diferenciais (IA governada, credenciais prontas) como gancho. |
| Custo de LLM descontrolado | Estouro de budget | Rate limit e atribuição por squad desde o dia 1. |
| Vazamento de dados via IA | Risco reputacional/legal | Prompt policy + mascaramento antes da persistência (regra arquitetural). |
| Lock-in de LLM ou registry | Risco estratégico | Abstração multi-LLM e multi-registry (regra arquitetural). |

### 6.1 Premissas a validar

- Engenheiros aceitam IDE web em vez de local (validar em entrevistas — ver task 1.1 entradas).
- Times aceitam workspaces efêmeros containerizados.
- Orçamento de LLM corporativo existe e é alocável por squad.

## 7. Regras e Políticas (herdadas do SpecDrive)

1. **Não incluir feature sem aderência a persona.** Toda nova task deve citar persona e job.
2. **Editor engine próprio fora do MVP.** Reutilizar motor existente (Monaco/Code-OSS).
3. **Separação control plane / execution plane.** Obrigatória na arquitetura (task 1.2).
4. **Sem lock-in** de LLM ou registry.
5. **Security-first.** Mascaramento de segredos antes de persistência.
